## 1. Repository Root

```
PIM-Monitor/
├── monitor-pipeline.yml         Pipeline definition (schedule, tasks, variables)
├── expected-changes.json        Optional: change suppression rules (user-managed)
├── inventory/                   PIM state snapshots (committed by the pipeline)
├── src/                         PowerShell source scripts
├── docs/                        Developer documentation (this directory)
├── docs-site/                   User-facing documentation site (Docusaurus)
└── .gitignore
```

| File / Folder | Owner | Purpose |
|---|---|---|
| `monitor-pipeline.yml` | Developer | Defines schedule, authentication task, and git commit step |
| `expected-changes.json` | Operator | Suppresses known-good changes to prevent alert fatigue |
| `inventory/` | Pipeline | Written and committed by the pipeline during each scan |
| `src/` | Developer | PowerShell scripts that execute during a scan run |
| `docs/` | Developer | This documentation |
| `docs-site/` | Developer | Docusaurus user documentation (separate from this) |

> [!NOTE]
> `expected-changes.json` is optional and managed by the operator, not generated by the pipeline. The pipeline reads it at the start of each scan and rewrites it (removing consumed and expired entries) after the filtering step.

## 2. src/

```
src/
├── Scan-PimState.ps1       Orchestrator — entry point for every pipeline run
├── graphEndpoints.ps1      URI constants and per-item URI builder functions
├── helpers.ps1             Shared utilities (pagination, JSON, inventory I/O)
├── diff.ps1                Diff engine and severity classification
├── git.ps1                 Git commit, push, and history functions
├── notifications.ps1       Email and webhook notification delivery
└── README.md               Testing guide for local validation
```

### Scan-PimState.ps1

The top-level orchestrator. It is the only file the pipeline executes directly. All other modules are dot-sourced from this file.

Responsibilities:
- Import modules in dependency order (`helpers.ps1` first).
- Acquire the Graph API access token via `Get-AzAccessToken`.
- Orchestrate all scan sections: lookups, activation events, Directory Roles, PIM Groups, expiring assignments.
- Filter expected changes, summarize results, and dispatch to git and notifications.

### graphEndpoints.ps1

Centralizes all Graph API URIs. Contains:
- `$script:GraphV1` and `$script:GraphBeta` base URL constants.
- `$script:GraphEndpoints` hashtable for collection-level endpoints (no per-item filter required).
- URI builder functions (`Get-RolePolicyUri`, `Get-GroupEligibleAssignmentsUri`, etc.) for per-item endpoints that require filtering by ID.

> [!TIP]
> All API version decisions live here. When Microsoft promotes a beta endpoint to v1.0, update only this file.

### helpers.ps1

General-purpose utilities used across all other modules:

| Function | Purpose |
|---|---|
| `ConvertTo-DeterministicJson` | Serialize any object to sorted, normalized JSON |
| `Get-AllGraphItems` | Paginate through a Graph endpoint, return all items |
| `Get-InventorySlug` | Convert `displayName` to kebab-case folder name |
| `New-InventoryFolder` | Create `inventory/{workload}/{slug}` if not exists |
| `Save-InventoryFile` | Write object to `{folder}/{name}.json` via deterministic serialization |

### diff.ps1

The change detection and classification engine. Functions:

| Function | Purpose |
|---|---|
| `Compare-InventoryFolder` | Main entry point — compares a folder against new data |
| `Compare-FlatProperties` | Property-level diff for `definition.json` files |
| `Compare-PolicyRules` | Rule-level diff for `policy.json` files |
| `Compare-Assignments` | Entry-level diff for `assignments.json` files |
| `Get-PolicyRuleSeverity` | Prefix-matched severity lookup for policy rule IDs |
| `Get-PropertySeverity` | Prefix-matched severity lookup for definition property names |
| `Get-AssignmentKey` | Stable composite key for assignment matching |
| `Remove-AssignmentNoise` | Strip volatile heartbeat fields before diff and write |
| `Find-ExpiringAssignments` | Detect assignments expiring within the configured window |
| `Test-ChangeIsExpected` | Match a change against `expected-changes.json` entries |
| `Test-ObjectEqual` | Compare two objects by normalizing both to deterministic JSON |
| `Test-ObjectHasKey` | Safe property presence check for both hashtables and PSObjects |
| `Read-PreviousInventoryFile` | Read and parse an existing inventory file from disk |
| `Group-ChangesBySeverity` | Aggregate change entries into severity buckets |

### git.ps1

Handles all git operations within the pipeline:

| Function | Purpose |
|---|---|
| `Publish-InventoryChanges` | Stage, commit, push; rebase once if push is rejected |
| `Get-StagedChanges` | Return name-status list for staged files |
| `Get-InventoryFileFromGit` | Read a file at a specific git ref (for historical diffing) |

### notifications.ps1

Notification delivery and payload construction:

| Function | Purpose |
|---|---|
| `Send-EmailNotification` | Compose HTML email and deliver via Graph sendMail |
| `Send-WebhookNotification` | Auto-detect webhook type and POST the payload |
| `Format-ChangeSummaryHtml` | Render full HTML email body with collapsible change cards |
| `Format-ChangeSummaryText` | Plain-text summary for logging and generic webhooks |
| `Export-ScanReport` | Write HTML report to disk for pipeline artifact publishing |
| `Build-TeamsPayload` | Adaptive Card JSON for Teams (via Power Automate webhook) |
| `Build-SlackPayload` | Block Kit JSON for Slack incoming webhooks |
| `Build-DiscordPayload` | Embed JSON for Discord channel webhooks |
| `Get-WebhookType` | Detect Teams / Slack / Discord / generic from URL |
| `Get-CommitDiffUrl` | Build ADO or GitHub commit diff URL from environment variables |
| `Select-ChangesForNotification` | Filter changes to those meeting minimum severity threshold |

## 3. inventory/

```
inventory/
├── directory-roles/
│   └── {role-slug}/
│       ├── definition.json
│       ├── policy.json
│       └── assignments.json
├── pim-groups/
│   └── {group-slug}/
│       ├── definition.json
│       ├── policy.json
│       └── assignments.json
├── authentication-contexts/
│   └── {slug}/
│       └── definition.json
├── administrative-units/
│   └── {slug}/
│       └── definition.json
└── activation-events/
    └── YYYY-MM.json
```

This directory is **not edited by humans**. It is written exclusively by the pipeline via `Save-InventoryFile`. Editing these files manually may produce spurious diffs on the next scan run.

The full schema for each file type is documented in [05-inventory-format.md](05-inventory-format.md).

> [!WARNING]
> Do not add `inventory/` to `.gitignore`. The pipeline relies on the checked-out inventory files as the previous state. If the folder is excluded, every scan will report every entity as new.

## 6. File Naming Conventions

| Convention | Example | Rule |
|---|---|---|
| Inventory folder names | `global-administrator` | kebab-case; derived from `displayName` via `Get-InventorySlug` |
| Inventory file names | `definition.json` | Always one of: `definition.json`, `policy.json`, `assignments.json` |
| Activation events files | `2026-04.json` | ISO year-month format; one file per calendar month |
| Source scripts | `Scan-PimState.ps1` | PascalCase; PowerShell verb-noun convention |
| Pipeline file | `monitor-pipeline.yml` | kebab-case; prefix `monitor-` to distinguish from other project pipelines |

## 7. File Relationships

```
monitor-pipeline.yml
    │
    └─ executes ──────────────── src/Scan-PimState.ps1
                                        │
                      dot-sources ──────┼──── src/helpers.ps1
                                        │         ↑
                      dot-sources ──────┼──── src/graphEndpoints.ps1
                                        │
                      dot-sources ──────┼──── src/diff.ps1
                                        │         (calls ConvertTo-DeterministicJson
                                        │          from helpers.ps1)
                      dot-sources ──────┼──── src/git.ps1
                                        │
                      dot-sources ──────┼──── src/notifications.ps1
                                        │         (calls ConvertTo-DeterministicJson
                                        │          from helpers.ps1)
                                        │
                         reads/writes ──┴──── inventory/
                                              (committed back to git by pipeline)
```

The orchestrator (`Scan-PimState.ps1`) is the only file that calls functions from multiple modules. Individual modules are cohesive and do not call each other, with one exception: `diff.ps1` and `notifications.ps1` both call `ConvertTo-DeterministicJson` from `helpers.ps1`.
